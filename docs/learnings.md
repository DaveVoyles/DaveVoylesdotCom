# Learnings

## 2026-07-21 — Sidebar bio text + avatar photo (PR #84)

- **This sandbox has no access to the user's real local filesystem on either of his machines, even though paths look plausible.** Dave twice gave a local file path he'd just saved a photo to (`/Users/davevoyles/Downloads/Dave_Debbie.jpeg`, then `/Volumes/davevoyles/Downloads/Dave_Debbie.jpeg` on "the correct" Mac) — both resolved to a real-looking but different filesystem here (`/Users/davevoyles/` exists and lists plausible-looking content, `Downloads/` has only 2 unrelated files, `/Volumes/davevoyles` doesn't exist at all). **Generalize: never assume a user-given local path is reachable just because the directory structure looks familiar — verify with `ls`, don't infer from a convincing-looking username/hostname match.**
- **Personal cloud-storage share links (OneDrive `1drv.ms` at minimum) are not fetchable by either `curl` or `WebFetch`.** `curl` hits Microsoft's WAF directly ("The request is blocked" HTML response, no image bytes). `WebFetch` chases the same redirect chain (`1drv.ms` → `photos.onedrive.com` → `onedrive.live.com/?qt=allmyphotos...`) into a JS-rendered SPA it can't execute, so it never reaches actual content. **Loading the same link in a real connected browser (claude-in-chrome) does work** — the photo renders — but the image itself loads client-side as a `blob:` object URL, and the extension's `javascript_tool` explicitly refuses to return base64-encoded blob data (`[BLOCKED: Base64 encoded data]`), a deliberate exfiltration guard, not a bug to route around. The browser's own "Download" button was the next thing tried, but this sandbox's Bash tool and the browser extension don't appear to share a filesystem either (no downloaded file ever appeared in any Bash-reachable directory). **The channel that actually worked: ask the user to upload the file directly into the target GitHub repo via the web UI (drag-and-drop into a file browser, a commit, or an issue/PR comment), then `git fetch`/pull it.** For any future "get me this specific personal file" ask where the user isn't local, lead with that path rather than re-attempting local-path or cloud-share delivery — it cost three back-and-forth rounds with Dave to arrive at it here.
- **A static asset the user uploads at full camera resolution (3.5MB, 4032x3024 from an iPhone) needs local resizing before it belongs in a `static/` (not `assets/`) path**, since `static/` is served byte-for-byte with zero Hugo image processing — unlike `assets/`, which gets Hugo's `resources.Get`/`.Resize`/`.Fill` pipeline for free. `sips` (pre-installed on macOS, no dependency to add) handled a clean center-crop-to-square + downsize + quality pass in three commands, landing at 82KB/480x480 for a 160px CSS-wide avatar slot — good enough without reaching for `assets/`+Hugo image processing for a single one-off image.
- **Both review-lenses passes on this PR caught something real, at each pass.** First pass: both `code-quality` and `testing` independently flagged the avatar `<img>`'s alt text being hardcoded to the site author's name even though the new photo shows two people — a real accessibility bug, not a nitpick, since the photo's content changed but the template's assumption ("this photo is a solo headshot of the site author") never got revisited. Fixed with an optional `homeInfoParams.AvatarAlt` config (mirroring the existing optional-`Avatar`-with-fallback pattern) rather than hardcoding new copy into the template. Second pass, after both fixes, came back clean (`Ready to Merge`, zero findings) — the gate isn't rubber-stamping, a genuinely clean diff really does synthesize to nothing.

## 2026-07-21 — Search-input disabled-until-`load` bug fix (PR #83)

- **A `<disabled>` form control cleared inside a `window` `load` listener is a real, reproducible bug, not just theoretical.** PaperMod's vendored `fastsearch.js` only sets `sInput.disabled = false` inside a callback registered on `window.addEventListener('load', ...)`, which waits for every subresource on the page — not just the DOM — before firing. Reproduced live on davevoyles.com: typing into the search box immediately after navigating dropped every keystroke (input silently stayed disabled), while typing again a few seconds later worked and returned real results. Since the script tag already carries `defer`, the DOM is guaranteed parsed by the time it runs — waiting for `load` on top of that bought nothing but a race window. **Generalize: a script loaded with `defer` never needs `window.addEventListener('load', ...)` to safely touch the DOM — that combination is a code smell worth flagging on sight.**
- **A pinned git submodule bug gets fixed by overriding the same-path asset at the site level, not by editing the submodule's working tree.** Hugo's asset-mount lookup prefers a site-level `assets/js/<name>.js` over the theme's `assets/js/<name>.js` at the same relative path — this repo's clean way to patch vendored `themes/PaperMod` behavior without diverging from the pinned upstream commit or fighting `git submodule` state. Left a header comment naming the exact upstream commit (`154d006e...`) the fork was taken from and what to do if PaperMod's own file changes later — both `code-quality` and `testing` review lenses independently flagged the missing provenance marker as a silent-drift risk before that comment existed, which is exactly the kind of convergent-independent-finding signal the review-lenses skill's own docs call out as a confidence booster.
- **This sandbox's Chrome browser-automation tool cannot reach `127.0.0.1` or `localhost` at all** — every navigation to either resolves to `chrome-error://chromewebdata/` (confirmed via `location.href` after the tool reported a "successful" navigate), while the same tab reaches public internet origins (`davevoyles.com`) fine. This is a different failure mode than the previously-documented cross-machine-localhost gap (2026-07-18's Console-theme entry, where the extension just runs on a different physical machine) — here the extension and the `hugo server`/`python -m http.server` process were on the *same* machine, and it still couldn't connect. Verify a local build's JS behavior via direct bundle grep (does the fingerprinted output contain the expected call) instead of assuming a local dev server is reachable from this environment's browser tool.
- **The computer-use `type` action can silently drop an entire typed string if it fires before the preceding `click` action's focus has actually landed in the renderer**, independent of any page-level `disabled` bug — hit this artifact twice in this session even *after* the real site bug was fixed (confirmed via JS: `disabled` was `false` and `document.activeElement` was already the right element, yet `.value` stayed empty). Inserting an explicit `wait` between `click` and `type` made it land reliably both times it was tried. Worth a beat of skepticism before concluding a fix didn't work from an automation-tool interaction alone — cross-check with a direct JS `.value` read, and retry once with a short wait, before trusting a single "nothing happened" reading of the tool as ground truth about the page.

## 2026-07-18/19 — Plan 0003 (post connections graph view) grilling + orchestration

- **A JS library's real data-shape contract is worth grepping the actual bundle for, not inferring from a plausible naming convention.** D1's build-time output was named `{nodes, edges}` (a perfectly reasonable, readable JSON schema) and D2's client-side code passed that object straight into `force-graph`'s `.graphData()` call — which silently fails, because the library's real (and only ever documented as) property name is `links`, never `edges`. `grep -o '\.links\b\|\.edges\b' force-graph.min.js` against the actual pinned bundle showed 13 hits for `.links`, zero for `.edges` — that one grep would have caught this before any code shipped. Instead it merged in D2 (#51) and shipped to production for roughly 20 minutes (the gap between that merge and D3's self-review catching it) with a completely blank `/graph/` page — the surrounding `try/catch` swallowed the `TypeError` to `console.error`, so neither the build nor a superficial "does the page return 200" check would ever catch it. **Generalize: when wiring hand-authored data into a third-party rendering library, grep the library's actual bundle/source for the exact property names it reads, don't trust that a semantically-equivalent name (edges≈links) will just work.**
- **A headless Node harness (jsdom + canvas) can drive a real browser-only JS library well enough to reproduce and disprove a runtime crash, when no connected browser tool is available.** No `claude-in-chrome`/computer-use tool was connected this session, which looked like a hard verification ceiling for a canvas-rendered interactive graph. `npm install jsdom canvas`, stub `global.window`/`document`/`requestAnimationFrame`, `require()` the actual pinned UMD bundle (not `eval` — the bundle's own `typeof module` check routes it to CommonJS export when required properly, but silently exports nothing useful under a bare `eval`), and call `.graphData()` with both the broken and fixed shapes — this reproduced the exact `TypeError` for the broken shape and confirmed clean initialization for the fixed one, empirically, against the real library. **Generalize: "no browser tool connected" doesn't mean "no way to execute the JS" — for libraries that only need a DOM + canvas surface (not a full browser engine), a headless jsdom+canvas harness is a legitimate, fast substitute for confirming a fix actually works, not just that it reads correctly.**
- **A denylist of known special-page layout names is a standing invitation for the next special page to leak through it.** D1's graph-data template excluded pages by checking `.Layout != "archives"` and `.Layout != "search"` — a reasonable filter until D2 added a third special page (`/graph/`, `layout = "graph"`) that the denylist had no way to know about, and it promptly appeared as a self-referential node in its own graph. The fix — `where site.RegularPages "Type" "posts"`, an allowlist keyed on the one property that's actually true of every real post and false for every root-level special page — can't be defeated by a future special page the same way, because it doesn't need updating when one is added.

## 2026-07-18 — Plan 0002 (related-posts/topics) orchestration + execution

- **Hugo treats a literal `/` inside a taxonomy term value as a URL hierarchy separator, not a display character.** The plan's originally-grilled curated bucket names ("AI / Agents", "Public Speaking / Presentations", "Career / Students", "Journalism / Marketing / PR") built and rendered fine through D1-D4 (mapping script, Related Content, the widget) because none of those deliverables ever turn a topic value into a URL path segment. D5 (auto-generated `/topics/<bucket>/` archive pages) is what actually exercises Hugo's taxonomy-term-to-path logic, and it split each slash-bearing bucket into nested directories (`/topics/career-/-students/` instead of one clean page) — Hugo's hierarchical-taxonomies feature (any `/` in a term is intentionally treated as parent/child nesting) firing on a name that was only ever meant to be a display string. Caught immediately by actually building and listing `public/topics/`, not by any template/config review — nothing about the config itself looked wrong. **Generalize: before locking in a curated controlled-vocabulary string that will become a Hugo taxonomy term, grep it for `/` — a punctuation choice that reads fine in a design doc's prose isn't automatically URL-safe once it becomes routing data.** The fix (renaming to slash-free forms, e.g. "Career and Students") required touching every downstream artifact already built on the old strings (a lookup table, its tests, 34 already-applied posts' front matter, 2 docs) — a good example of why constants meant to become URL path segments deserve that check up front, before a mapping script has already written them into 75 files.
- **A joiner character that fixes a URL problem can create a prose problem instead.** The first rename pass used "&" (`Career & Students`), which slugified to an ugly double-hyphen (`career--students`) since Hugo strips `&` entirely rather than converting it to a hyphen — cosmetically broken but not nested/wrong. The second pass switched to "and"/", ...and" joins, which slugify cleanly, but a hand-assembled code-quality review lens then flagged that the one 3-way bucket's comma form ("Journalism, Marketing and PR") read ambiguously wherever it sits inside an *outer* comma-separated list of all 6 bucket names (CONTEXT.md, the plan doc) — the internal comma isn't visually distinguishable from the list's own separators. Fixed by dropping the comma entirely ("Journalism and Marketing and PR"). Three iterations to get one naming decision right, each one only surfacing via either a real build or an independent review pass — neither would have caught what the other did.
- **`treehouse get --lease` keys its worktree pool off the *cwd's* apparent repo identity, not necessarily the name you'd expect.** Calling it from inside this session's own top-level worktree (itself nested under a differently-named parent, `.claude/worktrees/<random-name>`) leased a pool directory named after that wrapper, not `DaveVoylesdotCom` — cosmetically confusing (a `~/.treehouse/trusting-faraday-887c3a-.../` path for a `DaveVoylesdotCom` repo) but functionally correct (`git remote -v` inside it still resolved to the right repo). Verify the leased worktree's actual remote before trusting the pool-directory name at a glance, especially when leasing from inside another wrapper worktree rather than the plain repo root.
- **A stray `cd <primary-checkout> && git pull` during close-out is an easy isolation slip to make** — the orchestration loop's per-deliverable worktree discipline (claim → lease → branch → PR → teardown) doesn't automatically carry over to the close-out phase's doc updates, which feel like "just editing a couple of markdown files" and tempt a shortcut straight into the shared primary checkout. Caught before any actual edit landed there (only a fast-forward `pull`, not a commit), but the fix — lease one more worktree for the close-out commit too — should have been the default from the start, not a correction.

## 2026-07-18 — Platform-improvement pass (README, tags/nav, Console theme)

- **Content-derived tag backfills need to check the site's own existing vocabulary first, not just be internally consistent.** Copying a post's `categories` value straight into `tags` (e.g. `"Javascript / HTML5"`) looks fine in isolation but silently fragments Hugo's `/tags/` taxonomy when the rest of the site already uses atomic terms (`"JavaScript"`, `"HTML5"`) for the same topic — the compound tag gets its own dead-end page instead of merging into the existing one. Caught by the code-quality review lens, not by the build (a fragmented taxonomy still builds and 200s cleanly; only a real diff against existing `tags=` lines across the corpus would show it). Fix: `git grep -h '^tags' content/posts/*.md` before inventing any new tag string.
- **A CSS theme override needs to be checked against every page *template*, not just the homepage.** The Console theme's first pass styled post-list/post-single markup correctly but completely missed the Archives page (own class set: `archive-year-header`/`archive-entry-title`/`archive-meta`, nothing in common with `.post-entry`) and Search (results rendered client-side by `fastsearch.js` as bare `<li>` under `.searchResults`, no shared class either). A `hugo build` succeeding and a homepage screenshot both looked fine; only grepping the actual rendered `/archives/index.html` for the classes the CSS targeted surfaced the gap. Generalize: for any override/redesign that's meant to be site-wide, enumerate the distinct page *templates* (`themes/PaperMod/layouts/*.html`) first and verify the change reaches each one's real output, not just the page you happened to preview.
- **A CSS/design PR's self-review should include an actual contrast check, not just an eyeballed palette.** Computed WCAG contrast ratios (simple luminance formula, no external tool) for every fg/bg pair in both light and dark themes before merging the Console theme and caught one real borderline pair (tag-pill text at 3.81:1, below the 4.5:1 AA floor) that "looks fine" review would have missed.
- **Cross-machine browser automation (claude-in-chrome) can't reach a `localhost` dev server** if the connected Chrome extension is running on a different physical machine than the one running `hugo server` — the loopback address only resolves on the machine serving it. Verified visually instead by building the real site with `hugo --gc --minify`, extracting the actual compiled HTML + CSS, inlining the CSS into the HTML, and publishing that as an Artifact (a real rendered snapshot, not a hand-authored mockup) — works regardless of which machine the browser extension is attached to.

## 2026-07-19 — Graph hairball fix, homepage sidebar/grouping

- **A shipped "it works" feature can still be a real usability failure — quantify before redesigning.** Plan 0003's `/graph/` view passed all its acceptance criteria (valid JSON, renders, pan/zoom/click work) but Dave's reaction ("just looks like a whole web of things") was correct: computing actual edge density from `content/posts/*.md` showed 42.5% of all possible post-pairs had an edge, with hub posts touching 62-76% of the graph. Generalize: when a user calls a shipped, spec-passing feature impractical, measure the actual data shape before assuming the interaction/rendering layer is at fault.
- **Hugo's `complement COLL1 COLL2` returns elements in the LAST collection not in the first** — i.e. `complement $excludeList $realList` gives `$realList` minus `$excludeList`, not the other way around. Easy to get backwards; verified correct here by hand-checking a specific post-pair that shares only the excluded term against the built `graph-data.json`, not just trusting the template compiled without error.
- **Go/Hugo template variable reassignment (`$x = val`, not `:=`) works across nested `range`/`if` scopes back to an outer-declared variable** — used for both a "track the previous item across a loop" pattern (homepage year-grouping) and a "sum a weight across a range" pattern (graph edge weight), replacing a `newScratch` object that was originally reached for out of habit. Simpler and correct; no need for Scratch when the accumulator only needs to survive within one template's own execution.
- **A paginated list's per-page grouping needs to look backward across the page boundary, not just reset per page.** Naively resetting a "last seen group" variable at the top of each paginated page (mirroring how the *unpaginated* Archives page groups by year) produces a duplicate header when a group (year) spans two pages — caught by the `testing` review lens, not the build (both pages render individually valid HTML; only comparing consecutive pages' output reveals the duplicate). Fix: seed the tracking variable from the last item of the *previous* page (found by locating the current page's first item within the full, globally-sorted page list) rather than from a blank value.
- **Externally-facing prose (a public bio) still needs the human-confirm step even when the user gave the wording in-chat** — AGENTS.md's exception for generated externally-facing content exists precisely so a live public-page bio doesn't go through the same auto-approve path as an internal template fix; flagged explicitly in the PR body and this repo's HANDOFF rather than folding it into the same auto-merge-eligible bucket as the rest of the diff.

## 2026-07-19 — Plan 0004 (sidebar, sticky TOC, Vault section) orchestration

- **Goldmark never creates a link AST node for bare `[[Title]]` syntax — a render hook alone cannot intercept it.** `layouts/_default/_markup/render-link.html` only fires for nodes the CommonMark parser already recognizes as links (`[text](dest)`, autolinks, etc.); `[[Title]]` has no special meaning to CommonMark, so it's just literal text and the hook never sees it. Confirmed empirically with a probe test file before building around it. The working pattern is two-part: a raw-content regex pre-transform (`[[Title]]` → `[Title](<Title>)`, converting the custom syntax into a real markdown link the parser *will* recognize) piped through `.RenderString` (which does still respect render hooks, unlike raw `.Content`), then the render hook resolves the destination. **Generalize: before designing a render hook for any custom inline syntax, verify the parser actually emits an AST node for it — don't assume "render hooks intercept markdown rendering" covers syntax the parser itself doesn't recognize.**
- **CommonMark link destinations can't contain unescaped spaces unless wrapped in angle brackets** — `[Title](Multi Word Title)` silently fails to parse as a link (falls back to literal text, zero warning), while `[Title](<Multi Word Title>)` works. Directly bit the wikilink pre-transform above: multi-word note titles need the angle-bracket wrap or they silently never link. Easy to miss because single-word titles work fine without it, masking the bug until a real multi-word title is tested.
- **Hugo normalizes taxonomy plural identifiers to lowercase internally, even when `hugo.toml` declares a mixed-case plural.** Declaring `vaultTopic = 'vaultTopics'` in the taxonomy config, then calling `.GetTerms "vaultTopics"` in a template, silently returns zero results — Hugo stores and looks up the key as `vaulttopics`. No error, no warning; the page just renders with an empty topics list. Caught by dumping the actual data structure into a visible `<pre>DEBUG...</pre>` tag (HTML comments get silently stripped by Go's `html/template`, so debug output needs to land somewhere that actually renders). **This exact bug already existed, unnoticed, in the pre-existing `tags`/`topics` lookups in `layouts/single.html`** — it only "works" there by coincidence because those two taxonomy names happen to already be all-lowercase. Left undisturbed and documented in HANDOFF.md as a latent landmine for whoever next touches that file.
- **review-lenses caught a real bug on almost every one of 9 PRs this session** — a layout regression that would've silently affected all 76 existing posts, a site-wide broken taxonomy lookup, a title-collision silent-wrong-link bug, a `www.`-prefixed-URL misclassification, duplicate backlink rows, and a graph double-edge/dangling-edge risk. None of these were exotic — every one was the kind of thing a same-session author reviewing their own diff tends to gloss over precisely because they already know what the change was "supposed" to do (the skill's own stated rationale for why independent lenses beat single-pass self-review). Reinforces: don't skip or shortcut the gate even on a solo-orchestrated plan with no second engineer in sight.
- **A PR merged manually by the user mid-session, before a same-branch follow-up fix commit can ride along, is cleanly recoverable by cherry-picking the fix onto a fresh branch from the new `origin/main` and opening a second PR** — no content is lost, no force-push or history rewrite needed. Worth normalizing as the default recovery move rather than treating it as a surprise each time (the orchestrate skill's Supersession section covers *racing* sessions; this is the closely related "human merged out from under an in-flight fix" case, same underlying recovery shape).
