## 2026-07-18 — Platform-improvement pass (README, tags/nav, Console theme)

- **Content-derived tag backfills need to check the site's own existing vocabulary first, not just be internally consistent.** Copying a post's `categories` value straight into `tags` (e.g. `"Javascript / HTML5"`) looks fine in isolation but silently fragments Hugo's `/tags/` taxonomy when the rest of the site already uses atomic terms (`"JavaScript"`, `"HTML5"`) for the same topic — the compound tag gets its own dead-end page instead of merging into the existing one. Caught by the code-quality review lens, not by the build (a fragmented taxonomy still builds and 200s cleanly; only a real diff against existing `tags=` lines across the corpus would show it). Fix: `git grep -h '^tags' content/posts/*.md` before inventing any new tag string.
- **A CSS theme override needs to be checked against every page *template*, not just the homepage.** The Console theme's first pass styled post-list/post-single markup correctly but completely missed the Archives page (own class set: `archive-year-header`/`archive-entry-title`/`archive-meta`, nothing in common with `.post-entry`) and Search (results rendered client-side by `fastsearch.js` as bare `<li>` under `.searchResults`, no shared class either). A `hugo build` succeeding and a homepage screenshot both looked fine; only grepping the actual rendered `/archives/index.html` for the classes the CSS targeted surfaced the gap. Generalize: for any override/redesign that's meant to be site-wide, enumerate the distinct page *templates* (`themes/PaperMod/layouts/*.html`) first and verify the change reaches each one's real output, not just the page you happened to preview.
- **A CSS/design PR's self-review should include an actual contrast check, not just an eyeballed palette.** Computed WCAG contrast ratios (simple luminance formula, no external tool) for every fg/bg pair in both light and dark themes before merging the Console theme and caught one real borderline pair (tag-pill text at 3.81:1, below the 4.5:1 AA floor) that "looks fine" review would have missed.
- **Cross-machine browser automation (claude-in-chrome) can't reach a `localhost` dev server** if the connected Chrome extension is running on a different physical machine than the one running `hugo server` — the loopback address only resolves on the machine serving it. Verified visually instead by building the real site with `hugo --gc --minify`, extracting the actual compiled HTML + CSS, inlining the CSS into the HTML, and publishing that as an Artifact (a real rendered snapshot, not a hand-authored mockup) — works regardless of which machine the browser extension is attached to.
